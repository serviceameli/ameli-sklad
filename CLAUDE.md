## Архитектура

### Терминология
- **Визит** = один приезд человека на склад (может содержать несколько заказов)
- **Смена** = от нажатия «Начать смену» до «Закрыть смену» (кладовщик сам управляет)
- **Заказ** = конкретный номер из таблицы сайта

### Листы Google Sheets
| Лист | Назначение |
|------|-----------|
| `заказы` | Экспорт с сайта (название строчными буквами — чувствительно к регистру!) |
| `log` | Журнал всех визитов кладовщиков |
| `workers` | Список кладовщиков |
| `drafts` | Черновики незакрытых смен |

### Колонки листа `заказы` (экспорт сайта)

```
0  = Номер заказа
1  = Источник
2  = Получение, дата     (строка дд.мм.гггг)
3  = Получение, время    (формат "09:00 - 18:00")
4  = Возврат, дата
5  = Возврат, время
6  = Статус сайта        (для менеджеров — НЕ используется в логике склада!)
16 = Клиент
18 = Компания
19 = Работник            (если заполнен → «Наша доставка», иначе «Самовывоз»)
```

> ⚠️ **ВАЖНОЕ ПРАВИЛО**: колонка `Статус` (G, индекс 6) в листе `заказы` — это данные
> для менеджеров с сайта. Система учёта склада её **не читает** и **не использует**
> для определения что показывать кладовщику. Функция `updateOrderStatuses` пишет туда
> "Выдан"/"Возвращён" как побочный эффект для менеджеров, но логика склада строится
> исключительно на **датах** и **логе визитов**.

### Откуда берётся статус заказа в интерфейсе кладовщика

**Шаг 1 — Что показывать?** → только даты из листа `заказы`:
- `issueDate == сегодня` → показать как «К выдаче»
- `returnDate == сегодня` (и issueDate ≠ сегодня) → показать как «К возврату»
- Любая другая дата → не показывать

**Шаг 2 — Уже обработан?** → читаем лог визитов (`getProcessedOrderIds`):
- Если ID заказа есть в логе с операцией «Выдача» / «Получение (наш)» → помечаем как выданный (убираем из списка)
- Если есть с операцией «Возврат» / «Возврат (наш)» → помечаем как возвращённый

Функция `getProcessedOrderIds` сканирует **весь** лог (без фильтра по дате) — это безопасно,
потому что ID заказов уникальны, и `getTodayOrders` всё равно возвращает только заказы
с датой = сегодня.

**Ответ `getData`** возвращает `processedOrders: {issued:[...], returned:[...]}` —
фронтенд кладёт это в `serverIssuedToday` / `serverReturnedToday`.
Плюс текущая сессия кладовщика добавляется через `getSessionStatuses()` для мгновенного отклика.

## Ключевая логика

### Какие заказы показывать кладовщику
Статус из сайта **игнорируется**. Логика только по датам:

| Условие | Показываем как |
|---------|---------------|
| `issueDate = сегодня` | К выдаче |
| `returnDate = сегодня` (issueDate ≠ сегодня) | К возврату |
| `issueDate < сегодня` + заказ не в логе | 🔴 Просрочена выдача |
| `returnDate < сегодня` + заказ не в логе | 🔴 Просрочен возврат |
| Заказ есть в логе (любой день) | Не показывать |
| Будущая дата | Не показывать |

### Обработанные заказы
`getProcessedTodayIds()` читает лог за всё время и возвращает:

```javascript
{
  issued: [...],      // выданные сегодня (для счётчиков)
  returned: [...],    // возвращённые сегодня (для счётчиков)
  allIssued: [...],   // все выданные за всё время (чтобы не показывать просроченными)
  allReturned: [...], // все возвращённые за всё время
}
```

### Мультизаказы
Один визит может содержать несколько заказов. В логе — **одна строка на заказ**, но все строки одного визита имеют одинаковые: дату, время, кладовщика, операцию.

### Время
Всё время московское (UTC+3). Константа `MSK_OFFSET = 3`.
Ночная смена: с 22:00 до 9:00.

### Автообновление
При старте смены — немедленный запрос к серверу. Затем каждые 60 секунд пока смена открыта.

## Apps Script — эндпоинты

### GET
| action | Возвращает | Для кого |
|--------|-----------|----------|
| `getData` | `{workers, orders, processedToday}` | Кладовщик |
| `getAll` | `{shifts, orders}` | Дашборд |
| `getUnmatched` | `{unmatchedVisits, unlistedOrders}` | Сверка |

### POST
| Тело запроса | Действие |
|-------------|----------|
| `{isDraft: true, ...}` | Сохранить черновик смены |
| `{action: 'linkVisit', visitKey, orderIds[]}` | Связать визит с заказами |
| Основной (без action) | Записать закрытую смену в лог |

## Важные нюансы

1. **Имя листа `заказы` — строчными буквами.** Google Apps Script чувствителен к регистру.
2. **Даты в таблице** — строки `дд.мм.гггг`, парсятся через `parseDate()` в скрипте.
3. **После изменения Apps Script** — всегда новое развёртывание, URL меняется, нужно обновить `config.js`.
4. **`no-cors` режим** — основные POST-запросы используют `mode: 'no-cors'`, ответ не читается. Только `linkVisit` использует обычный fetch и ждёт ответа.
5. **GitHub Pages** — файлы опубликованы, ссылки вида `https://serviceameli.github.io/ameli-sklad/warehouse-staff.html`.
