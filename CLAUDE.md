## Архитектура логики

### Какие заказы показывать кладовщику — только по датам

| Условие | Отображение |
|---------|------------|
| `issueDate == сегодня` И `returnDate != сегодня` | К выдаче |
| `returnDate == сегодня` И `issueDate != сегодня` | К возврату |
| `issueDate == returnDate == сегодня` | **Однодневная аренда** — сначала К выдаче, после выдачи → К возврату |
| Заказ есть в логе → выдан | Убираем из «К выдаче» |
| Заказ есть в логе → возвращён | Убираем из «К возврату» |
| Просроченные даты + не в логе | 🔴 Просрочен |

> ⚠️ **ВАЖНО**: Колонка `Статус` в листе `заказы` — для менеджеров сайта, склад её НЕ читает.
> Статус определяется ТОЛЬКО по датам + логу визитов.

### Однодневная аренда (`sameDay: true`)
Заказ с `issueDate == returnDate == today` получает флаг `sameDay: true`.
На фронтенде функция `getEffectiveOrderType(o, issuedIds, returnedIds)`:
- не выдан → `'issue'`
- выдан, не возвращён → `'return'`
- возвращён → `'done'`

### Обработанные заказы
`getProcessedOrderIds()` сканирует **весь** лог (без фильтра по дате).
Возвращает `{issued:[...], returned:[...]}` — ID заказов из лога.

`getData` возвращает `processedOrders` → фронтенд кладёт в `serverIssuedToday` / `serverReturnedToday`.
Плюс текущая сессия через `getSessionStatuses()` для мгновенного отклика.

### Операции в логе
Операции: **Выдача** / **Возврат** / **Выдача и возврат** (однодневная).
Старые значения: «Получение (наш)», «Возврат (наш)», «Получение+Возврат» — обратная совместимость сохранена.

## Apps Script — эндпоинты

### GET
| action + параметры | Возвращает |
|---|---|
| `getData` | `{workers, orders, processedOrders}` |
| `getData&worker=Имя` | + `draft` (черновик этого кладовщика) |
| `getAll` | `{shifts, orders}` для дашборда |

### POST actions
| `action` в теле | Действие |
|---|---|
| `isDraft: true` | Сохранить черновик в лист `drafts` |
| `action: 'addVisit'` | **Немедленно** записать один визит в лог |
| `action: 'closeShift'` | Удалить строки этой смены + перезаписать все визиты |

## Страница кладовщика (`warehouse-staff.html`)

### Персональные ссылки
Каждый кладовщик имеет ссылку: `warehouse-staff.html?worker=Имя+Фамилия`

**Логика при открытии:**
- Нет черновика → **Welcome-экран**: «Открыть смену» / «Посмотреть заказы»
- Черновик есть, < 12 часов → сразу **рабочий экран** (смена восстанавливается)
- Черновик есть, > 12 часов → экран **«Закрыть старую смену»** / «Продолжить»
- Без параметра `?worker=...` → обычный экран выбора кладовщика

### Три вкладки
1. **＋ Визит** — форма добавления визита (кто приехал, какие заказы, время)
2. **📦 Заказы** — статусы всех заказов дня (обновляется каждые 60 сек, видят все кладовщики)
3. **📋 Смена** — журнал визитов смены + кнопка «Закрыть смену»

### Ключевые функции
- `addVisit()` → пишет в лог немедленно через `sendVisitNow()` (action: addVisit)
- `saveDraft()` → localStorage под ключом `wh_draft_ИмяФамилии` (per-worker)
- `getEffectiveOrderType(o, issuedIds, returnedIds)` → тип заказа с учётом sameDay
- `getSessionStatuses()` → объединяет серверные данные + текущая сессия
- `refreshOrdersFromServer()` → каждые 60 сек обновляет заказы

### Важные детали
- `localStorage` ключ per-worker: `wh_draft_Имя_Фамилия` — кладовщики не конфликтуют
- PIN-система закомментирована (отключена), код сохранён для будущего включения
- Даты в `sendVisitNow` и `confirmSubmit` → явный формат `dd.MM.yyyy` через MSK-хелперы
- `deleteShiftRows` сравнивает ISO-даты через `.toISOString()` (Sheets хранит их как Date объекты)

## Технические нюансы

1. **`no-cors` режим** — POST-запросы fire-and-forget, ответ не читается
2. **Даты в Sheets** — ISO-строки автоматически конвертируются в Date объекты при записи
3. **Имя листа `заказы`** — строчными, чувствительно к регистру
4. **Даты парсятся** через `parseDate()` — поддерживает `дд.мм.гггг`, `yyyy-mm-dd`, Excel serial
5. **После деплоя Apps Script** — URL меняется, обновить в `config.js` и закоммитить
