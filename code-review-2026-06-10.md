## 🔴 Критические баги — исправить в первую очередь

### 1. Вкладка «Сверка» в дашборде не работает: эндпоинтов нет в Apps Script
Дашборд вызывает `?action=getUnmatched` (строки 506, 1078) и POST `action:'linkVisit'` (строка 1189), но в `google-apps-script.js` **нет ни одного из этих обработчиков**. Последствия:

- `getUnmatched` возвращает `{error:'Unknown action'}` → вкладка всегда показывает «Всё в порядке», бейдж не появляется.
- Хуже: POST `linkVisit` в `doPost` проваливается в ветку **closeShift** (она без проверки action) → вызывается `deleteShiftRows(log, undefined, undefined)` и `log.appendRow` с пустыми/undefined значениями → **в лог пишется мусорная строка**, плюс `clearDraft(undefined)`.

**Правка:**
1. В `doPost` сделать ветку closeShift явной: `if (data.action === 'closeShift') {...}` и на неизвестный action возвращать ошибку.
2. Либо дописать `getUnmatched`/`linkVisit` в Apps Script, либо убрать вкладку «Сверка» из дашборда.
3. Проверить: возможно, эти функции есть в задеплоенной версии, а файл в репо отстал — кстати, URL в `config.js` **не совпадает** с «актуальным» URL в CLAUDE.md. Синхронизировать.

### 2. Удаление визита с несколькими заказами стирает только одну строку лога
`writeEntryToLog` пишет **по одной строке на каждый заказ** визита. `deleteVisitRow` (строка 246) делает `return` после первого совпадения. Если в визите было 2–3 заказа, после удаления в логе остаются строки-сироты → заказы продолжают числиться выданными/возвращёнными, через 15 сек refresh вернёт их в счётчики у кладовщика.

**Правка:** убрать `return`, удалять **все** строки с совпадающими `shiftStart + worker + timeAuto` (цикл уже идёт с конца, индексы не съедут).

### 3. Дашборд: заказы «К возврату» отображаются как «✓ Выдан»
`statusOf` (warehouse-dashboard.html, ~567) проверяет сначала `issued.has(o.id)`. Для многодневного заказа, который выдали неделю назад и сегодня ждут возврат, `issued` содержит его ID (лог сканируется целиком) → статус 'issued' → бейдж «✓ Выдан» и заказ уезжает в секцию «Выдано/Возвращено», хотя возврат ещё не принят. При этом счётчик сверху считает его ожидающим — данные противоречат друг другу.

**Правка:**
```js
const statusOf = o => {
  const t = effectiveType(o);              // учитывает sameDay
  if (t === 'return') return returned.has(o.id) ? 'returned' : 'pending';
  if (t === 'done')   return o.sameDay || o.type==='return' ? 'returned' : 'issued';
  return issued.has(o.id) ? 'issued' : 'pending';
};
```

### 4. Сравнение дат строками `dd.MM.yyyy` ломается на границе месяца
`getOverdueOrders`: `issueDate < today`, `returnDate < today`, `m.returnDate < today` (строки 364, 367, 413) сравнивают **строки** в формате `dd.MM.yyyy`. `"31.05.2026" < "10.06.2026"` → `false`, потому что `"3" > "1"`. Итог: заказы, просроченные с прошлого месяца, **не попадают в просрочку**, а в начале месяца возможны ложные срабатывания.

**Правка:** сравнивать в формате `yyyy-MM-dd` (`Utilities.formatDate(..., 'yyyy-MM-dd')`) или как Date-объекты.

### 5. Триггер просрочки в 00:00 помечает заказы НОВОГО дня
`recordOverdueOrders` проверяет `issueDate === today && !issuedSet.has(...)` → «Просрочен - не выдан». Триггер создаётся на `atHour(0)` (00:00–01:00 МСК). В полночь `today` — это уже новый день: все заказы наступившего дня сразу получают запись «Просрочен», хотя день ещё не начался. Пока заказ остаётся в листе `заказы`, дедупликация в `getData` маскирует проблему, но лог и журнал дашборда засоряются ложными записями, и комментарий в коде («23:00») противоречит реальному времени.

**Правка:** либо `atHour(23)` и оставить `=== today`, либо оставить `atHour(0)` и проверять `=== вчера`.

## 🟠 Существенные замечания

### 6. Операция «both» не учитывается как выдача на фронте
`getSessionStatuses` (staff, ~1207): `['issue','pickup']` → issued, `['return','dropoff','both']` → returned. Визит с операцией «Выдача и возврат» не помечает заказ выданным в текущей сессии. На бэке (`getProcessedOrderIds`) «both» корректно попадает в оба набора — фронт и бэк расходятся. То же в дашборде: `w.issued` не учитывает both, `w.returned` учитывает (строки 868–869), и в `renderStats` (issuedO).

**Правка:** добавить `'both'` в список выдачи во всех трёх местах.

### 7. getAllShifts: слияние смен и визитов, фантомный «Система»
- Ключ смены `дата_работник_деньНочь`: две смены одного кладовщика в один день (утром и вечером, обе «День» по началу) сольются в одну → табель часов посчитает по first shiftStart/shiftEnd.
- Ключ визита `время_посетитель_операция`: два разных визита в одну минуту с одинаковым типом склеятся.
- Строки «Просрочен …» от «Система» создают группу-смену → в «Кладовщиках» и «Табеле» появляется карточка «Система» с 0 часов.

**Правка:** включить `shiftStart` в ключ смены; в ключ визита добавить `timeAuto`; строки с worker `Система` пропускать (`if (r[3] === 'Система') return;`).

### 8. Атрибуция визитов по дате/дню недели через shiftStart
В `getAllShifts` `timestamp: r[1]` — это **начало смены**, не время визита. Дашборд везде использует `v.ts`:
- ночная смена через полночь — все визиты падают на дату начала смены;
- `ts` — UTC: смена, начатая в 00:30 МСК, в статистике уедет на предыдущий день;
- `new Date(v.ts).getDay()` в тепловой карте — день недели в **локальной таймзоне браузера** зрителя.

**Правка:** строить дату визита из `shiftDate` (колонка 0) + `v.time`, день недели считать по МСК. Для визита после полуночи в ночной смене — сдвигать дату на +1 день, если `v.time < время начала смены`.

### 9. Дашборд не показывает просрочку
`getAll` возвращает только `getTodayOrders()`. Кладовщик просрочку видит, руководитель — нет. Добавить в `getAll` overdue-заказы (как в `getData`) и секцию «🔴 Просрочены» в `renderOrders`.

### 10. confirmLink: fetch с `Content-Type: application/json` не пройдёт CORS
Обычный (не no-cors) POST с JSON вызывает preflight OPTIONS, который Apps Script не поддерживает → запрос упадёт ещё в браузере. Когда будете дописывать `linkVisit`: отправлять с `Content-Type: text/plain` (Apps Script всё равно парсит `e.postData.contents`), тогда preflight не нужен и ответ можно прочитать.

### 11. Нет блокировок в doPost — гонки записи
Одновременные `closeShift` (удаление + перезапись строк) и `addVisit` от другого кладовщика могут пересечься: индексы строк сдвигаются между чтением и `deleteRow`. **Правка:** обернуть тело `doPost` в `LockService.getScriptLock()` с `waitLock(20000)`.

### 12. День/Ночь визита фиксируется по моменту записи
`addVisit`: `night: isNight()` — текущее время, а не выбранное время визита. Если кладовщик вносит визит задним числом (поле времени редактируется), флаг будет неверным. Также `sendVisitNow` шлёт `isNight` смены от текущего момента, а `closeShift` потом перезапишет по началу смены — пока смена не закрыта, в логе и статистике значения «плавают». **Правка:** считать `night` от `visit.time`, а `isNight` смены — всегда от `shiftStart` (как в confirmSubmit).

## 💡 Улучшения (не баги)

- **Надёжность доставки визитов.** Все POST — `no-cors` fire-and-forget: если у кладовщика моргнул интернет, визит молча теряется до закрытия смены. Предложение: очередь неотправленных визитов в localStorage с повтором при следующем онлайне + индикатор «не отправлено: N».
- **Квоты Apps Script.** `getData` сканирует весь лог (`getProcessedOrderIds`) при каждом poll каждые 15 сек с каждого открытого устройства. С ростом лога станет медленно и упрётся в квоты. Варианты: кэш `CacheService` на 15–30 сек на стороне Apps Script; и/или поднять интервал до 30–60 сек.
- **Авто-обновление дашборда** — сейчас данные грузятся один раз при открытии; добавить refresh раз в 1–2 мин или кнопку.
- **Архивирование лога** — раз в N месяцев переносить старые строки в лист `log_архив`, чтобы скан оставался быстрым.
- **Один источник логики статусов.** `getEffectiveOrderType` продублирован в staff и (в упрощённом виде) в дашборде — расхождение уже породило баг №3. Вынести в общий `shared.js`.
